## プロジェクト概要

これは、LLMベースの文書抽出精度を検証するためのLLMOps精度検証プラットフォームです。既存のRailsプロダクトからLLM機能を分離し、精度実験を実行するためのFastAPIベースのサービスを提供します。

## アーキテクチャ

本プロジェクトは、厳格な4層アーキテクチャを持つドメイン駆動設計（DDD）に従っています：

1. **インターフェース層** (`src/interfaces/`) - APIエンドポイントとリクエスト/レスポンススキーマ
2. **アプリケーション層** (`src/application/`) - ユースケースとデータ転送オブジェクト（DTO）
3. **ドメイン層** (`src/domain/`) - コアビジネスロジック（外部依存なし）
4. **インフラストラクチャ層** (`src/infrastructure/`) - 外部サービスと永続化

**重要なルール:**
- ドメイン層は他の層に依存してはいけない
- 依存関係は内向き: Interface → Application → Domain ← Infrastructure
- インフラストラクチャサービスには依存性注入を使用
- 層間のデータ転送にはDTOを使用

## LLMOps開発フロー

### 要件受領時の標準作業手順

ユーザーから要件を受け取った際、AIは以下の手順に従って作業を進める：

#### 1. 要件分析フェーズ
```
1. 要件の種類を判定
   - 新機能追加
   - 既存機能の変更
   - バグ修正
   - リファクタリング
   
2. 影響範囲の特定
   - 関連するドメイン定義を特定
   - 影響を受ける機能/画面/APIを特定
```

#### 2. ドキュメント確認フェーズ
```
1. 現状確認（必須）
   - /docs/domain/配下の関連ファイルを読み込み
   - /docs/architecture/metadata.yamlで依存関係を確認
   - 影響を受けるapplication/interfaces/を確認

2. 変更要否の判定
   - ドメイン定義の変更が必要か
   - 新規ドキュメントが必要か
```

#### 3. 設計・更新フェーズ
```
1. ドメイン層の更新（必要な場合）
   - domain/models/: エンティティ、集約、値オブジェクトの追加/変更
   - domain/rules/validation-rules.md: バリデーションルールの追加/変更
   - domain/rules/business-rules.md: ビジネスルールの追加/変更

2. 他層の更新（必要な場合）
   - application/use-cases/: ユースケースの追加/変更
   - interfaces/api/contracts.md: API仕様の追加/変更
   - infrastructure/database/schema.md: DBスキーマの追加/変更

3. metadata.yamlの更新（必須）
   - 新しい依存関係を追加
   - 影響範囲を記録
```

#### 4. 検証フェーズ
```
1. ドキュメント整合性チェック
   - 参照リンクの有効性確認
   - 必須セクションの存在確認
   - 層間の依存関係確認

2. 変更影響の最終確認
   - metadata.yamlの影響範囲を再確認
   - 実装への影響を明示
```

#### 5. 実装フェーズ
```
1. Python型定義の生成
   - ドメイン定義から自動生成（手動作成禁止）

2. 実装コードの作成
   - ドメイン定義に厳密に従う
   - DRY原則を遵守
```

### ファイル作成ルール

#### domain/配下
- **原則**: 既存ファイルの更新を優先
- **新規作成条件**: 新しいドメイン概念が追加される場合
- **必須内容**: ユビキタス言語、ビジネスルール

#### application/配下
- **新規作成条件**: 新しいユースケースが追加される場合
- **命名規則**: `ユースケース名.md`（kebab-case）
- **必須内容**: ドメイン定義への参照リンク

#### interfaces/配下
- **新規作成条件**: 新しいAPIや画面が追加される場合
- **命名規則**: APIはkebab-case、画面はsnake_case
- **必須内容**: ドメイン定義への参照

#### infrastructure/配下
- **新規作成条件**: 新しい技術コンポーネントが追加される場合
- **必須内容**: 技術詳細、ドメイン層とのマッピング

## 過去の失敗事例と対策（2025/07/18追加）

### 事例1: 命名統一不備によるインポートエラー
**問題**: `FieldResult` → `FieldEvaluationResult`の変更時、テストファイル内の古いクラス名が残存
**対策**: クラス名変更時は必ず以下を実行
```bash
# 全ファイル検索で影響範囲確認
grep -r "FieldResult" src/ tests/
# 一括置換後の再確認
grep -r "FieldResult" src/ tests/
```

### 事例2: Docker環境とローカル環境の差異
**問題**: ローカルでテスト成功後、Docker環境でインポートエラー発生
**対策**: 重要な変更後は必ずDocker環境テストを実行
```bash
docker-compose build
docker-compose run --rm app pytest tests/unit/ -v
```

### 事例3: 段階的テスト不足による後戻り
**問題**: 実装完了後のエンドツーエンドテストで大量修正が必要
**対策**: 実装中の段階的テスト実行
```bash
# 各実装ステップ後に実行
mypy src/                    # 静的解析
pytest tests/unit/           # ユニットテスト  
docker-compose run --rm app pytest  # Docker環境テスト
```
